Why Governance

复杂系统不是“缺智能”
而是缺结构

缺边界、缺审计、缺可验证的决策权 —— 于是系统依赖“人记得”，最终被技术债吞没。

🔍

隐形依赖

为了效率绕过入口与意图校验，形成“看不见的路径”。它不会立即爆炸，只会慢慢腐蚀系统。

🛡️

权限墙裂缝

未授权导出往往不是“没写权限”，而是缺少强制入口与审计证据链：调用路径绕行，系统无法证明自己。

⏮️

无法回放

出了问题只能猜：谁在什么意图下、走了哪条路径？没有 trace 与审计，就没有治理，更谈不上合规。

The Constitution

让系统的每一次行为
可被制度解释

SynOSX 的价值不在“多做事”，而在“把判断权制度化”。

三铁律（核心契约）

Traceability（可追溯）

每一步都有记录：能定位、能回放、能复盘。依靠 trace_id + 审计链路闭环。

Expressibility（可表达）

规则与意图被写成可校验的契约，不靠口头约定：制度有唯一公理真源。

Growth（可进化）

系统能长期成长而不腐烂：版本对齐、边界清晰、依赖合法；坏边可以被工具裁决并阻断。

MUST 宪法级硬约束

  • 所有关键行为必须带 intent 与 audit trace_id，并可回放。
  • 制度契约是唯一真源：规则先行，执行层只负责校验与解释。
  • 依赖链必须保持 DAG：任何绕过治理链路的路径都应被自动化裁决。

MUST NOT 违宪路径（腐蚀风险）

  • 业务模块绕过入口直接调用内部实现（形成 Invisible Path）。
  • 为赶进度“临时魔改”绕过 Intent/Firewall，导致审计证据链断裂。
  • 把“可信”当成口头承诺，而非通过证据链进行技术化验证。

裁决流程（NL → Intent → Firewall → Audit → Execution → Report）

DECISION FLOW
自然语言 / 外部请求 → Intent 生成（可校验） → Firewall 权限裁决 → Audit 证据记录 → 业务模块执行 → Report / 可回放报告

系统不接受匿名、越权或未被认可的发声源； AI 与外部系统仅能通过被制度认可的入口提出意图； 治理中枢负责裁决与记录，业务模块只执行被允许的请求。 解释者、裁决者、执行者从不越界——系统因此可长期演进。

S0–S6 Families

治理运行时（S0–S6）

明确职责的器官：SynShell / Intent / Firewall / Audit / Dictionary / NLP / Persona。

🔧

SynShell（S0）

运行环境宪法：自检、runtime、health checks，保证底座一致性。

🎯

Intent（S1）

把“想做什么”变成可校验的意图契约；禁止绕过入口，禁止隐形路径。

🛡️

Firewall（S2）

权限墙与边界守卫：谁能做、能做多少、在什么上下文可被允许。

🧾

Audit（S3）

事实记录与证据链：为回放、复盘、RefleX 与合规准备基础数据。

📚

Dictionary（S4）

语义与领域字典：统一术语、字段与域边界，降低“同名不同义”的结构腐蚀。

🧠

NLP / Persona（S5–S6）

自然语言 → Intent；人格接口只决定交互风格，不越权处理领域数据。

Use Cases

落地场景（可验证）

不是“我们能做什么”，而是“我们如何让行为可证明、可回放”。

🧾

开票合规 & 输出治理

导出/开票必须携带 intent + 权限裁决 + 审计打点，避免“节前临时绕过”留下隐形后门。

🚪

权限墙（Firewall）

把“门”变成制度：任何导出、跨域访问都必须通过守卫与审计，违宪路径自动可见、可阻断。

🗺️

AI 语义入口

AI 不直接写业务数据：只生成 intent；治理中枢负责裁决与记录；模块只执行合法请求。
